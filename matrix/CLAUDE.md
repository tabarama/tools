# CLAUDE.md

Этот файл читается агентом Claude Code перед каждым действием в репозитории. Здесь — постоянные инструкции: стек, структура, конвенции. Подробная доменная и продуктовая логика — в `matrix.md`.

## Контекст проекта

Инструмент для финансового аналитика в POS-кредитовании (РФ). Управляет матрицами комиссионного вознаграждения (reward), из которых собираются сценарии для расчёта фактического и гипотетического КВ по сделкам за месяц.

Полная спецификация: **`matrix.md`**. Эта ссылка — обязательная, без неё любое изменение будет «в сторону вообще». Перед началом задачи — прочти `matrix.md` целиком.

## Технологический стек

- Python 3.11+
- Streamlit (UI)
- SQLAlchemy 2.x + Alembic (ORM + миграции)
- SQLite (хранилище, файл `app.db` в рабочей директории пользователя)
- openpyxl (чтение/запись xlsx)
- pyodbc (DWH через ODBC DSN)
- pandas (обработка данных — только там, где переиспользуется legacy-код)
- pytest (тесты)

## Правила работы с кодом

### Что можно и нужно

- **Переиспользовать функции из `legacy/`** — импортируй `evaluate_groups`, `build_rules_index`, `build_agreement_index`, `clean_text`, `load_month_from_dwh` и т.д.
- Писать типизированный код (type hints на сигнатурах публичных функций)
- Писать docstring на публичные функции (одна строка минимум — что делает)
- Логировать через `logging` с именем модуля
- Валидировать входные данные через Pydantic там, где данные пересекают границу модуля

### Что нельзя

- **Не редактировать файлы в `legacy/`.** Если понадобится доработать логику — делай обёртку в `src/adapters/`, а не меняй legacy
- Не менять бизнес-логику расчёта КВ. Если кажется, что в legacy есть баг — сообщи и жди подтверждения, не исправляй молча
- Не использовать глобальные переменные для состояния (только Streamlit session state и передача через параметры)
- Не делать сетевых запросов, кроме DWH через pyodbc
- Не добавлять зависимости без согласования (особенно: Django, Flask, FastAPI — не нужны; React, Vue — не нужны)

### Конвенции

- Имена файлов и директорий: `snake_case`
- Имена классов: `PascalCase`
- Имена функций/переменных: `snake_case`
- SQL-таблицы: `snake_case` (единственное число: `template`, не `templates`)
- Строки в БД: все в lowercase после нормализации (bank_name, proc_type хранятся уже нормализованными)

## Работа с legacy-кодом

Ключевой принцип: **новый инструмент — это оркестратор поверх legacy**. Logic остаётся в legacy, новый код отвечает за:
1. UI
2. Хранение шаблонов/сценариев в SQLite
3. Сборку DataFrame правил и договоров в том формате, что ожидает legacy
4. Вызов legacy-функций
5. Запись результата в xlsx

В `src/adapters/legacy_adapter.py` — все точки контакта. Если нужно вызвать что-то из legacy — только через adapter, напрямую не импортируй в pages/ или runner/.

## Тесты

- `tests/` — pytest
- Обязательный тест: `tests/test_regression.py` проверяет, что раннер даёт **побайтно** тот же xlsx, что `legacy/reward_matrix.py` на фикстурах
- Фикстуры — маленький синтетический xlsm + моковые данные DWH (сохранить как parquet или csv)
- Перед коммитом: `pytest` должен проходить

## Запуск

Локально:
```bash
streamlit run app.py
```

При первом запуске:
- создаётся `app.db` в текущей директории
- применяются миграции
- на странице Settings (если есть) или в `.env` — настроить DSN для DWH

2. Сформулируй план реализации Phase 1 (список задач с приоритетом)
